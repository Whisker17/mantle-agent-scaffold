## 1. Executive Summary

The Mantle MCP v0.1-core implementation **meets the design spec and release plan requirements** for the foundational milestone. Core tenets (T1–T6) are respected; tools, resources, server wiring, and error handling align with the specification. No critical or blocking issues were found. A few minor deviations, documentation gaps, and one unused dependency warrant follow-up before v0.2.

**Overall assessment:** Ready for next phase (v0.2-readplus) after addressing Important findings.

## 3. Findings by Severity

### Critical

*None.*

### Important

#### I1. `mantle_getTokenPrices` input validation too permissive

**Spec:** `tokens: z.array(z.string()).min(1)`  
**Implementation:** `tokens` treated as any array; empty array yields empty `prices` and `partial: true`.

**Location:** `src/tools/token.ts:122-126`

```typescript
const tokens = Array.isArray(args.tokens) ? args.tokens.map(String) : [];
```

**Impact:** Spec requires at least one token; empty input should be rejected with an error.

**Recommendation:** Add input validation and throw `MantleMcpError` for empty `tokens`:

```typescript
if (tokens.length === 0) {
  throw new MantleMcpError(
    "INVALID_INPUT",
    "At least one token is required.",
    "Provide one or more token symbols or addresses.",
    { field: "tokens" }
  );
}
```

---

#### I2. Unused `zod` dependency

**Location:** `package.json:27`

`zod` is listed as a dependency but never imported. The design spec uses zod notation for documentation only; implementation relies on manual checks.

**Impact:** Extra transitive dependency and bundle size; unclear if zod was intended for runtime validation.

**Recommendation:** Either (a) remove from `package.json` or (b) adopt zod for input validation across tools for consistency and stronger typing.

---

#### I3. Skill mapping example differs from tool API

**Spec Section 3.8:** Example shows `mantle_getAllowances(owner: "0xABC...", tokens: ["USDC"], spenders: [...], network: "mainnet")`.

**Actual tool schema:** `owner` + `pairs: [{ token, spender }]`.

**Location:** `specs/mcp-design.md` (around line 668)

**Impact:** Agents following the example would invoke the tool incorrectly.

**Recommendation:** Update the workflow example in the spec to use the `pairs` schema:

```
mantle_getAllowances(owner: "0xABC...", pairs: [{ token: "USDC", spender: "0x319B..." }], network: "mainnet")
```

---

#### I4. Tool dispatch is O(n) instead of O(1) as specified

**Spec Section 3.4:** "Flat tool merge... allTools lookup for O(1) dispatch" — shows `const tool = allTools[name]`.

**Implementation:** `src/server.ts:42-43`

```typescript
const tool = Object.values(allTools).find((item) => item.name === name);
```

This iterates all tools on every call. The root cause: `allTools` keys are short names (`getChainInfo`) while the lookup uses full names (`mantle_getChainInfo`).

**Impact:** With 10 tools the perf hit is negligible, but it will grow. More importantly it deviates from the spec's explicit O(1) contract.

**Recommendation:** Re-key `allTools` by `tool.name` in `src/tools/index.ts`:

```typescript
export const allTools: Record<string, Tool> = Object.fromEntries(
  [...Object.values(chainTools), ...Object.values(registryTools),
   ...Object.values(accountTools), ...Object.values(tokenTools)]
    .map((t) => [t.name, t])
);
```

---

#### I5. Tool descriptions lack concrete Mantle examples

**Spec Section 5.0:** "Tool descriptions MUST include concrete Mantle examples with real addresses."

**Implementation:** Most tool descriptions are terse. For example:

- `mantle_getBalance`: `"Get native MNT balance for an address."` (no example)
- `mantle_getTokenBalances`: `"Batch read ERC-20 token balances for an address."` (no example)
- `mantle_resolveAddress`: `"Resolve trusted contract addresses by key, alias, or label from the Mantle registry."` (no example)

The spec explicitly shows rich descriptions like:
```
"mantle_resolveAddress: Lookup a contract address by key, symbol, or alias...
Examples:
- "USDC" on mainnet → 0x09Bc4E0D864854c6aFB6eB9A9cdF58aC190D0dF9 (confidence: high)
- "Agni Router" on mainnet → 0x319B69888b0d11cEC22caA5034e25FfFBDc88421"
```

**Impact:** AI agents rely on tool descriptions to understand tool capabilities. Minimal descriptions reduce the agent's ability to use tools correctly without additional lookups.

**Recommendation:** Enrich all tool descriptions with concrete Mantle examples as the spec requires.

### Minor

#### M1. `getChainInfo` returns extra fields not in tool spec

**Spec Output (5.1):** `chain_id`, `name`, `native_token`, `rpc_url`, `ws_url`, `explorer_url`, `bridge_url`, `recommended_solidity_compiler`

**Implementation:** Also returns `wrapped_mnt`, and for sepolia `faucet_urls`.

**Location:** `src/config/chains.ts`, `src/tools/chain.ts`

**Impact:** Additive; likely useful for clients. Spec and resources (6.1) already mention `wrapped_mnt` for chain resources.

**Recommendation:** Update tool output description in the spec to include `wrapped_mnt` and optional `faucet_urls` for clarity.

---

#### M2. `GetPromptRequestSchema` throws raw `Error`, not `MantleMcpError`

**Location:** `src/server.ts:91-94`

```typescript
if (!messages) {
  throw new Error(`Prompt not found: ${request.params.name}`);
}
```

**Impact:** MCP SDK will handle the error, but it will not follow the `{ error, code, message, suggestion, details }` shape. Low priority for v0.1 (empty prompts).

**Recommendation:** Consider throwing `MantleMcpError` with code `PROMPT_NOT_FOUND` for consistency, or document that prompt errors use SDK defaults.

---

#### M3. Missing `INVALID_INPUT` in error catalog

**Spec 8.3 Error Code Catalog:** No `INVALID_INPUT` code listed. Implementation (and recommended I1 fix) may need it.

**Recommendation:** Add `INVALID_INPUT` to the catalog if used for validation failures.

---

#### M4. `package.json` deviations from spec

**Spec 3.3:** Shows `"dev": "tsc --watch"` and `"prepublishOnly": "npm run build"` scripts.

**Implementation:** Missing both. `package.json` has `build`, `start`, `typecheck`, `test` only.

**Impact:** Development experience; `prepublishOnly` matters for npm publishing.

**Recommendation:** Add missing scripts before npm publish workflow.

---

#### M5. No `vitest.config.ts`

**Spec 3.7 (directory structure):** Lists `vitest.config.ts` in project root.

**Implementation:** Absent. vitest runs with default config from tsconfig.

**Impact:** Minor; vitest infers config. Explicit config would clarify test patterns and setup.

---

#### M6. `getTokenBalances` and `getAllowances` iterate sequentially

**Spec 5.3:** `mantle_getTokenBalances` — "Uses multicall for efficiency."

**Implementation:** `src/tools/account.ts:120-148` — sequential `for...of` loop calling `readTokenBalance` per token.

```typescript
for (const rawToken of tokens) {
  // ...
  const balanceRaw = await resolvedDeps.readTokenBalance(client, resolved.address, address);
}
```

Same pattern in `getAllowances` at line 174.

**Impact:** Misses the multicall efficiency the spec promises. With many tokens, latency compounds linearly.

**Recommendation:** Implement `Promise.allSettled` for concurrent reads, or use viem's `multicall` for batched RPC calls as the spec intends.

### Informational

#### N1. `prefetchResources` is no-op

**Location:** `src/resources.ts:82-84`

```typescript
export async function prefetchResources(): Promise<void> {
  return Promise.resolve();
}
```

**Impact:** None for v0.1; all resources are static. Intent is correct for future remote resources.

---

#### N2. Registry `environment` terminology

**Location:** `src/lib/registry.ts`, `registry.json`

Registry uses `environment: "mainnet" | "testnet"` while tools use `network: "mainnet" | "sepolia"`. `listRegistryEntries` correctly maps `sepolia` → `testnet`.

**Recommendation:** Add a short comment in `registry.ts` explaining the mapping for future maintainers.

---

#### N3. `token-list.ts` uses Node.js `node:crypto`

**Location:** `src/lib/token-list.ts:1`

```typescript
import { createHash } from "node:crypto";
```

**Impact:** None for v0.1 (stdio = Node.js only). If HTTP/edge transport is added later, this would need replacement with a Web Crypto API.

---

#### N4. No `src/abis/` directory structure

**Spec 3.7:** Shows `src/abis/` with `index.ts`, `erc20.ts`, `agni-router.ts`, etc.

**Implementation:** ERC20 ABI lives in `src/lib/erc20.ts`. No `abis/` directory.

**Impact:** Acceptable for v0.1 (only ERC20 ABI needed). Introduce the `abis/` structure in v0.2 when DeFi-specific ABIs are needed.

---

#### N5. `src/server.ts` wiring differs from spec skeleton

**Spec 3.4:** Shows all tool modules imported individually:
```typescript
import { chainTools } from "./tools/chain.js";
import { registryTools } from "./tools/registry.js";
// ... 8 more imports
```

**Implementation:** Cleaner — imports only `allTools` from `tools/index.ts`:
```typescript
import { allTools } from "./tools/index.js";
```

**Impact:** Positive deviation. The implementation is cleaner than the spec's example.

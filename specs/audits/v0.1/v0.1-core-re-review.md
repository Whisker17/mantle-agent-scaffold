## 2. New Findings (Introduced or Revealed by Fixes)

### NEW-1. Double import from errors.js in server.ts

**Location:** `src/server.ts:11-12`

```typescript
import { MantleMcpError } from "./errors.js";
import { toErrorPayload } from "./errors.js";
```

**Assessment:** Cosmetic. Same module imported twice; consolidate into:

```typescript
import { MantleMcpError, toErrorPayload } from "./errors.js";
```

**Severity:** Minor (code hygiene)

---

### NEW-2. `getChainStatus` description semantically misleading (I5 fix quality)

**Location:** `src/tools/chain.ts:93-94`

**Current:**
```
"Return live block height and gas price from Mantle RPC. Examples: mainnet status for USDC market context at 0x09Bc4E0D864854c6aFB6eB9A9cdF58aC190D0dF9; sepolia status for WMNT at 0x19f5557E23e9914A18239990f6C70D68FDF0deD5."
```

**Problem:** `getChainStatus` returns block height and gas price — it has no relationship to USDC or WMNT. The addresses were injected to satisfy the "must contain real address" requirement without being semantically meaningful. The spec (`mcp-design.md:886-889`) shows output examples: `block_number=12345678, gas_price_gwei="0.02", syncing=false`.

**Recommendation:** Replace with examples that illustrate output shape or use-case context, e.g.:
```
Examples: mainnet block_number=12345678, gas_price_gwei="0.02"; sepolia status before testing token interactions.
```

**Severity:** Minor (agent UX; misleading examples could confuse downstream reasoning)

---

### NEW-3. `getBalance` description example quality

**Location:** `src/tools/account.ts:225-227`

**Current:**
```
"Get native MNT balance for an address. Examples: address=0x1111111111111111111111111111111111111111; common gas-spend workflows include Agni router interactions at 0x319B69888b0d11cEC22caA5034e25FfFBDc88421."
```

**Problem:** `0x1111...1111` is a placeholder, not a real Mantle address. The spec requires "real addresses." Agni router (`0x319B...`) is a spender context, not the typical owner for balance queries. A real wallet or contract address and expected output would be more useful.

**Recommendation:** Use a real Mantle mainnet address (e.g. from registry) and show expected output format: `address=0x... → balance_mnt="1.5"`.

**Severity:** Minor (spec compliance; agent UX)

---

### NEW-4. `tool-descriptions.test.ts` regex too weak

**Location:** `tests/tool-descriptions.test.ts:9`

```typescript
expect(tool.description).toMatch(/0x[a-fA-F0-9]{8}/);
```

**Problem:** Matches only 8 hex chars after `0x` — any partial hex string passes. A valid Ethereum address has 40 hex chars (20 bytes). The weak regex allowed semantically incorrect or placeholder addresses (NEW-2, NEW-3) to pass.

**Recommendation:** Strengthen to full address format:
```typescript
expect(tool.description).toMatch(/0x[a-fA-F0-9]{40}/);
```
Then update descriptions that use placeholders or partial addresses to use real 40-char addresses.

**Severity:** Minor (test quality; gates spec compliance)

---

### NEW-5. `PROMPT_NOT_FOUND` not added to error catalog

**Location:** `specs/mcp-design.md` Section 8.3

**Problem:** M2 introduced `PROMPT_NOT_FOUND` in `server.ts`, but the error code catalog (lines 3153-3179) was only updated for `INVALID_INPUT`. `PROMPT_NOT_FOUND` is now part of the implementation but undocumented in the spec.

**Recommendation:** Add to error catalog:
```
| `PROMPT_NOT_FOUND` | Requested prompt does not exist | GetPromptRequestSchema with unknown name |
```

**Severity:** Minor (spec/documentation completeness)

## 4. Overall Assessment

### What Went Well

- **Important findings (I1–I5):** All structurally addressed. Input validation, dependency cleanup, spec alignment, O(1) dispatch, and description enrichment are in place.
- **Minor findings (M1–M6):** All addressed. Spec updates, error handling, scripts, config, and concurrent reads implemented correctly.
- **Test coverage:** New tests for I1 (empty token rejection) and I4 (O(1) keyed map) provide regression protection.
- **Architecture:** No regressions. DI, error handling, and tool dispatch remain sound.

### Gaps Identified

- **I5 quality:** Letter of the requirement met (examples with addresses), but some descriptions are semantically weak or use placeholders.
- **Test rigor:** `tool-descriptions.test.ts` regex does not enforce full 40-char address format.
- **Documentation:** `PROMPT_NOT_FOUND` missing from error catalog.

### Consensus Status

**Consensus reached with caveats.** The audit response’s fixes are implemented and verified. The new findings (NEW-1 through NEW-5) do not invalidate the fixes; they represent incremental improvements. The implementor (Codex 5.3) can either:

1. **Option A (recommended):** Address NEW-1 through NEW-5 in a small follow-up before v0.2, or  
2. **Option B:** Proceed to v0.2-readplus and track these as technical debt / doc fixes.

# Mantle MCP v0.2-readplus Implementation Audit

**Audit Date:** 2026-03-01  
**Implementation Scope:** v0.2-readplus (design spec + release plan Section 3)  
**Implementor:** Codex 5.3  
**Auditor:** Claude Opus (code-reviewer subagent + manual review)  
**References:** `specs/mcp-design.md`, `specs/mcp-release-plan.md`, `docs/plans/2026-02-28-mcp-v0-2-readplus.md`

---

## 1. Executive Summary

The Mantle MCP v0.2-readplus implementation delivers the structural scope of the release plan: DeFi read tools, indexer tools, diagnostics tools, endpoint safety policy, three prompts, and two new documentation resources. All 35 tests pass, typecheck is clean, and the modular architecture from v0.1 carries forward cleanly.

However, several spec behaviors are **not implemented or significantly condensed**:

- The `"best"` provider in `getSwapQuote` silently falls back to Agni instead of trying both DEXes and comparing quotes.
- Pool liquidity USD derivation from reserves + `getTokenPrices` is skipped entirely.
- Prompt content is substantially thinner than the spec's detailed multi-step workflows.
- The endpoint policy has an inconsistency where `http://127.0.0.1` is blocked while `http://localhost` is allowed under the same local-HTTP flag.
- The `querySubgraph` tool lacks the response-size cap required by the security model.
- All five v0.1 re-review findings (NEW-1 through NEW-5) remain unaddressed.

**Overall assessment:** Structurally complete; needs behavioral fixes (C1, I1–I5) before the v0.2 milestone can be considered fully compliant with the design spec.

## 3. Findings by Severity

### Critical

#### C1. `getSwapQuote` does not implement `"best"` provider routing

**Spec:** `provider: z.enum(["agni", "merchant_moe", "best"]).default("best")`  
**Spec Example (5.6):** "Swap 1 WETH → USDC via best: tries Agni and Merchant Moe, returns better quote"

**Location:** `src/tools/defi-read.ts:146-148`

```typescript
const provider: "agni" | "merchant_moe" =
  providerInput === "merchant_moe" ? "merchant_moe" : "agni";
```

**Problem:** When `provider="best"` (the default), the tool silently routes to Agni only. The spec explicitly requires querying both DEXes and returning the better quote. Since `"best"` is the default value, **most agent-initiated swap quotes will only see Agni prices**, potentially missing better Merchant Moe routes.

**Impact:** Agents using the default `"best"` provider will receive suboptimal quotes without knowing Merchant Moe was never consulted. This violates the tool's documented contract and could lead to worse execution for users.

**Recommendation:** When `provider === "best"`:
1. Call `quoteProvider` for both `"agni"` and `"merchant_moe"` (via `Promise.allSettled`).
2. Compare `estimated_out_raw` values.
3. Return the quote with the higher output.
4. If one provider fails with `NO_ROUTE`, return the other.
5. If both fail, throw `NO_ROUTE`.

```typescript
if (providerInput === "best") {
  const [agniResult, moeResult] = await Promise.allSettled([
    resolvedDeps.quoteProvider({ provider: "agni", ... }),
    resolvedDeps.quoteProvider({ provider: "merchant_moe", ... })
  ]);
  // pick the better quote or the one that succeeded
}
```

### Important

#### I1. `getPoolLiquidity` skips USD derivation from reserves + `getTokenPrices`

**Spec (5.6, Pool Liquidity USD valuation rule):**
> - Prefer protocol/indexer-provided USD liquidity if available.
> - Otherwise derive from reserves using `mantle_getTokenPrices`.
> - If both fail, return `total_liquidity_usd: null` with warning.

**Location:** `src/tools/defi-read.ts:278-281`

```typescript
const totalLiquidityUsd = typeof data.total_liquidity_usd === "number"
  ? data.total_liquidity_usd : null;
if (totalLiquidityUsd == null) {
  warnings.push("total_liquidity_usd is null due to unavailable valuation source.");
}
```

**Problem:** The implementation only checks the `readPool` provider's `total_liquidity_usd`. It never attempts the second-tier derivation (reserves × token prices). The tool jumps directly to the null fallback with a warning.

**Impact:** USD valuation will be null for any pool where the provider doesn't supply it, even when token prices are available from `getTokenPrices`. This reduces the utility of portfolio analysis and risk evaluation workflows.

**Recommendation:** Add a second-tier derivation step:
1. If `total_liquidity_usd` is null and both token decimals are known, call `getTokenPrices` for both tokens.
2. Derive: `reserve_0_decimal × price_0 + reserve_1_decimal × price_1`.
3. If derivation fails (prices unavailable), keep null and add the existing warning.
4. Inject `getTokenPrices` through the deps interface.

#### I2. `querySubgraph` has no response-size cap

**Spec Section 9.4:**
> "Request timeout and response-size caps to limit abuse and context exhaustion."

**Location:** `src/tools/indexer.ts:76-100`

**Problem:** The implementation has timeout (via `AbortController`) and row truncation (for SQL queries via `maxRows`), but `querySubgraph` has no response-size limit. A misconfigured or malicious subgraph could return an arbitrarily large JSON response, exhausting agent context or memory.

In contrast, `queryIndexerSql` correctly implements `maxRows` truncation with warnings.

**Impact:** Potential context exhaustion and agent degradation when querying verbose subgraphs.

**Recommendation:** Add a response-size cap:
- Accept a configurable `MANTLE_SUBGRAPH_MAX_RESPONSE_BYTES` (default ~1MB).
- Stream or check `content-length` before parsing.
- If exceeded, truncate and add `warnings: ["response truncated by size limit"]`.

#### I3. Endpoint policy: `http://127.0.0.1` blocked despite local-HTTP flag

**Spec Section 9.4:**
> "Optional local HTTP override: `http://` allowed only when explicitly enabled by operator config for local development."

**Location:** `src/lib/endpoint-policy.ts:64-82`

```typescript
// Line 65: Protocol check — passes for localhost, 127.0.0.1, ::1 when flag is true
if (url.protocol !== "https:" && !(allowHttpLocal && url.protocol === "http:" && isLocalHost(url.hostname))) {
  throw ...
}

// Line 75: Private IP check — blocks 127.0.0.1 and ::1 unconditionally
if (isPrivateIpv4(host) || isBlockedIpv6(host)) {
  throw ...
}
```

**Problem:** When `MANTLE_ALLOW_HTTP_LOCAL_ENDPOINTS=true`:
- `http://localhost:8080/graphql` → passes protocol check → `isPrivateIpv4("localhost")` is false → **ALLOWED**
- `http://127.0.0.1:8080/graphql` → passes protocol check → `isPrivateIpv4("127.0.0.1")` is true (127.x.x.x range) → **BLOCKED**
- `http://[::1]:8080/graphql` → passes protocol check → `isBlockedIpv6("::1")` is true → **BLOCKED**

These are functionally equivalent endpoints. The inconsistency will confuse operators configuring local development environments.

**Impact:** Operators using `127.0.0.1` or `::1` instead of `localhost` will hit `ENDPOINT_NOT_ALLOWED` despite enabling the local-HTTP flag.

**Recommendation:** When `allowHttpLocal` is true and the hostname is a loopback address (localhost, 127.0.0.1, ::1), skip the private-IP and blocked-IPv6 checks:

```typescript
const isLoopback = isLocalHost(host);
if (!isLoopback && (isPrivateIpv4(host) || isBlockedIpv6(host))) {
  throw new MantleMcpError("ENDPOINT_NOT_ALLOWED", ...);
}
```

#### I4. Prompt content significantly condensed vs. spec

**Spec Examples (7.1, 7.7, 7.8):**
- `mantle_portfolioAudit`: 7 steps with code examples, allowance pairs, risk classification rules, formatted report table, and Common Mistakes table (~80 lines)
- `mantle_mantleBasics`: Architecture section, Key Facts table, Core Tokens table, DeFi Landscape table, Getting Started steps (~50 lines)
- `mantle_gasConfiguration`: Gas Costs table, live gas price check, RPC Endpoints table, L1 Data Fee explanation, Common Mistakes table (~50 lines)

**Implementation** (`src/prompts.ts`):
- `mantle_portfolioAudit`: 5 numbered steps, no code examples, no tables, no risk classification (~8 lines)
- `mantle_mantleBasics`: 4 bullet points, no tables, no architecture details (~6 lines)
- `mantle_gasConfiguration`: 5 bullet points, no gas cost table (~6 lines)

**Impact:** Prompts are the primary agent guidance surface. Thin content means agents may produce incomplete workflows, skip critical steps (e.g., risk classification in portfolio audit, Common Mistakes), or hallucinate details that should come from the prompt.

**Recommendation:** Expand all three prompts to match the spec's level of detail. At minimum, include:
- Step-by-step workflow structure
- Concrete tool call examples with expected output shapes
- Tables for reference data (gas costs, tokens, protocols)
- Common Mistakes sections

#### M2. `getSwapQuote` identity-swap fallback for same-token pairs

**Location:** `src/tools/defi-read.ts:182-190`

```typescript
if (!quote && tokenIn.address.toLowerCase() === tokenOut.address.toLowerCase()) {
  quote = {
    estimated_out_raw: amountInRaw.toString(),
    estimated_out_decimal: amountInInput,
    price_impact_pct: 0,
    route: `${tokenIn.symbol}->${tokenOut.symbol}`,
    fee_tier: feeTier
  };
}
```

**Problem:** This is not in the spec. When token_in and token_out are the same, the tool returns a 1:1 quote instead of rejecting the request. No DEX would execute a same-token swap; this could mislead agents.

**Recommendation:** Reject same-token swaps with `INVALID_INPUT`:
```typescript
if (tokenIn.address.toLowerCase() === tokenOut.address.toLowerCase()) {
  throw new MantleMcpError("INVALID_INPUT", "token_in and token_out must be different.", ...);
}
```

#### M3. `querySubgraph` never populates `warnings` array

**Spec:** `warnings: string[]  # e.g., "response truncated by pagination limit"`

**Location:** `src/tools/indexer.ts:89`

```typescript
warnings: []
```

**Problem:** Always returns empty `warnings`. No logic detects pagination limits, large responses, or partial data. In contrast, `queryIndexerSql` correctly populates `warnings` for truncation.

**Recommendation:** At minimum, add pagination detection (if response contains `pageInfo` or similar indicators) and response-size warnings.

#### M4. `probeEndpoint` does not enforce method enum at runtime

**Spec:** `method: z.enum(["eth_chainId", "eth_blockNumber", "eth_getBalance"]).default("eth_blockNumber")`

**Location:** `src/tools/diagnostics.ts:107-108`

```typescript
const method =
  typeof args.method === "string" ? args.method : "eth_blockNumber";
```

**Problem:** The `inputSchema` correctly declares the enum, but runtime validation does not enforce it. Any arbitrary RPC method string is passed through, potentially allowing agents to invoke dangerous methods (e.g., `debug_*`, `eth_sendRawTransaction`).

**Recommendation:** Validate at runtime:
```typescript
const ALLOWED_METHODS = ["eth_chainId", "eth_blockNumber", "eth_getBalance"];
if (!ALLOWED_METHODS.includes(method)) {
  throw new MantleMcpError("INVALID_INPUT", `Method not allowed: ${method}`, ...);
}
```

#### M5. `getPoolLiquidity` reserve decimal calculation may throw untyped error

**Location:** `src/tools/defi-read.ts:290-293`

```typescript
reserve_0_decimal:
  data.token_0.decimals == null ? null : formatUnits(BigInt(data.reserve_0_raw), data.token_0.decimals),
```

**Problem:** If `reserve_0_raw` is not a valid BigInt string (e.g., corrupted pool data), `BigInt()` throws a raw `TypeError` instead of a typed `MantleMcpError`.

**Recommendation:** Wrap in try-catch and produce a `MantleMcpError`, or validate the raw string before conversion.

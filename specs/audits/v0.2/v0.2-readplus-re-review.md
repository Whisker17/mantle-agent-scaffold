## 2. New Findings (Introduced or Revealed by Fixes)

### NEW-R1. Subgraph response-size cap is post-hoc (not streaming)

**Location:** `src/tools/indexer.ts:52-55`

```typescript
const text = await response.text();
const responseBytes = Buffer.byteLength(text, "utf8");
const json = text.length === 0 ? {} : JSON.parse(text);
return { json, elapsed_ms: Date.now() - start, response_bytes: responseBytes };
```

**Problem:** The full response body is loaded into memory via `response.text()` before the byte-size check runs (at L120). For truly large responses (e.g., 50MB+), the memory exhaustion happens before the cap triggers. A streaming approach or `Content-Length` pre-check would avoid this.

**Assessment:** For v0.2 scope (read-only tools, controlled indexer endpoints), this is acceptable. The cap does prevent oversized responses from propagating into agent context, which was the primary concern. Streaming enforcement is a hardening improvement for v0.5.

**Severity:** Informational (v0.5 hardening scope)

**Severity:** Minor (content polish; not blocking)

### NEW-R3. `getChainStatus` and `getPoolLiquidity` descriptions omit addresses

**Location:** `src/tools/chain.ts:95`, `src/tools/defi-read.ts:546`

`getChainStatus` description no longer contains any addresses (previously had misleading ones). `getPoolLiquidity` description references token addresses but not a pool address. The spec says "Tool descriptions MUST include concrete Mantle examples with real addresses."

**Assessment:** For `getChainStatus`, removing misleading addresses was the correct fix (NEW-2). The tool outputs block height and gas price — addresses are not semantically relevant. For `getPoolLiquidity`, the description has USDC and USDT addresses, which is sufficient. The `tool-descriptions.test.ts` correctly validates format-when-present rather than mandating addresses on every tool.

**Severity:** Informational (no action needed)

## 4. Overall Assessment

### What Went Well

- **C1 (Critical fix):** The `"best"` provider routing is clean and correct — `Promise.allSettled`, candidate comparison, single-provider fallback with warning, and `NO_ROUTE` when both fail. This was the highest-priority fix and it's well-implemented.
- **I1 (USD derivation):** The 3-tier pool liquidity valuation matches the spec exactly. The `getTokenPrices` injection via deps keeps the DI pattern consistent.
- **I3 (Endpoint policy):** The `allowLoopbackHttp` variable eliminates the inconsistency cleanly. All edge cases verified.
- **M2-M5 (Minor fixes):** Same-token rejection, pagination warnings, method enforcement, and BigInt error handling are all well-implemented with corresponding tests.
- **v0.1 carry-forward items:** All five NEW-1 through NEW-5 items from the v0.1 re-review are addressed.
- **Test coverage:** Grew from 35 to 44 tests. New tests cover best-mode routing, single-provider fallback, same-token rejection, USD derivation, reserve parsing errors, loopback HTTP, response-size cap, pagination warnings, and method validation.
- **Spec updates:** Both `PROMPT_NOT_FOUND` and `MANTLE_SUBGRAPH_MAX_RESPONSE_BYTES` added to the design spec.

### Gaps Identified

- **I2 quality:** Response-size cap is functional but post-hoc (memory already consumed). Acceptable for v0.2; streaming approach recommended for v0.5.
- **I4 completeness:** Prompt structure is correct but content has residual gaps vs. spec (missing tokens, gas rows, L1 fee section). Non-blocking.
